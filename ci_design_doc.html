Tying Product Usage to Competitive Loss

01 Problem & Scope

When a deal is lost to a competitor, did the customer already have low usage of existing products? This pipeline joins deal outcomes with product usage data to answer that question — and to identify accounts heading the same direction before they churn.

In Scope

  • Closed Lost deals with competitor attribution (junction object)
  • Product usage at close date — point-in-time, not current state
  • Competitor win/loss analysis from CI_Opportunity_Competitor
  • Usage trend (12-month lookback from close date)
  • Loss reason classification from Competitive_Notes free text
  • At-risk identification on retained accounts matching the loss pattern

Out of Scope (V1)

  • Win analysis — V1 focuses on losses. Win patterns are a separate pipeline.
  • Product telemetry beyond CRM — no Mixpanel/Amplitude/Pendo integration
  • Real-time alerting — V1 is daily batch. Streaming is V2+.
  • Battlecard content generation — feedback loop is V2. V1 surfaces the data.
  • Predictive modeling — V1 is descriptive + diagnostic. ML is V3.

04 Design Decisions & Tradeoffs

Every decision below has alternatives. These are choices, not defaults. If the team has different priorities, the architecture accommodates them because these decisions are localized, not baked into the foundation.

D1. Point-in-Time Join vs Current State

CRITICAL
Option AJoin to current usage. Simplest. No snapshots needed. But every churned account shows 0% usage — you learn nothing about what usage looked like when the deal was active.
Option BJoin to usage as of close date (SCD Type 2). Requires daily snapshots. Captures the actual state when the decision was made. Reveals patterns invisible in current state.
ChosenOption B. Without historical state, the analysis is meaningless. This is the entire value proposition.
CostMust build and maintain SCD Type 2 snapshots. ~365 rows per account per year. If snapshots don’t exist historically, we only have precision from the day we start building them.
Reversible?Yes — can always fall back to Option A by removing the date filter. But we can’t go back to B from A (lost data).

D2. LEFT JOIN vs INNER JOIN

IMPORTANT
Option AINNER JOIN. Only include deals where usage data exists. Cleaner dataset. But silently drops deals where usage tracking wasn’t configured — biases analysis toward already-monitored accounts.
Option BLEFT JOIN. Keep ALL lost deals. NULL usage = tracking not configured = itself a blind spot signal worth surfacing.
ChosenOption B. The accounts we can’t see are often the ones most at risk. NULLs are data.
CostDownstream consumers must handle NULLs. Every metric calculation needs COALESCE or explicit NULL logic. Dashboard must display “No usage data” states.
Reversible?Yes — add WHERE clause to filter NULLs. No data loss.

D3. Snapshot Grain: Daily vs Monthly

Option ADaily snapshots. Most precise point-in-time join. Storage: ~365 rows × accounts/year.
Option BMonthly averages. 12× smaller. But a deal that closed on the 5th gets the full-month average — could be misleading if usage dropped mid-month.
Option CSingle row per account. Simplest join. Loses time dimension entirely. Useless for trend analysis.
ChosenOption A for the join table (precision). Pre-aggregated monthly summaries for the dashboard (performance).
CostLarger table. If 100K accounts × 730 days (2yr) = ~73M rows. Manageable for Snowflake but needs partitioning on Snapshot_Date.
Reversible?Can always aggregate daily → monthly. Cannot disaggregate monthly → daily.

D4. Quality Gate: Hard Block vs Soft Warn

Option AHard block. If critical QA check fails, pipeline stops. Serve stale data. Alert team.
Option BSoft warn. Always refresh. Flag issues in metadata. Risk: bad data reaches dashboard before anyone notices.
ChosenOption A for schema validation and uniqueness (hard block). Option B for completeness and format checks (warn, proceed). Bad data in a dashboard is worse than stale data.
CostDashboard may show stale data during outages. Requires “Last refreshed” timestamp visible to users. Ops team needs alerting channel.
Reversible?Yes — change a config flag per check.

D5. Loss Reason Extraction

Option ALLM / zero-shot classification on Competitive_Notes. Automated. Scales. But accuracy is unknown until tested on real data. Could produce garbage on short or vague notes.
Option BUse existing Primary Loss Reason picklist. Already there. But Rob said it’s not granular enough for CI analysis.
Option CHybrid: use picklist as baseline, LLM as enrichment. Picklist value always present. LLM adds granularity where notes exist. Human review on low-confidence classifications.
ChosenOption C. Ship with picklist (reliable). Add LLM classification as a second column. Don’t replace the source data — augment it.
CostTwo columns to maintain. LLM inference cost per run. Need accuracy baseline before trusting automated output.
Reversible?Yes — LLM column is additive. Can drop it without affecting core pipeline.

D6. Ingestion Frequency

Option ADaily batch from Snowflake. Simple. Data already lands there. Latency: ~24hr worst case.
Option BCDC (Change Data Capture) for Opportunities. Near-real-time on deal closes. More complex. Deals close in bursts (quarter-end), not steadily.
ChosenOption A for V1. Deal outcomes don’t change after close. Usage shifts slowly. Daily batch is 10× simpler and sufficient for weekly/monthly CI analysis. CDC is V2 if latency becomes a business problem.
CostDashboard is always ~24hr behind. Acceptable for strategic CI work. Not acceptable for real-time CS triggers (V2 problem).

D7. Multi-Competitor Attribution

Option AKeep only Leading competitor. Cleaner: one row per deal. But loses context — you don’t know who else was in the evaluation.
Option BKeep all competitors per deal. Grain: one row per deal-competitor. Deal Amount appears on multiple rows — must deduplicate in aggregations (SUM Amount WHERE Leading = TRUE).
ChosenOption B. Richer analysis. Use Leading flag to filter for winner in aggregations. The non-leading competitors tell you who was considered — valuable for competitive landscape mapping.
CostEvery aggregate query must filter on Leading = TRUE to avoid double-counting deal amounts. Documentation critical. Easy to get wrong.

05 Pipeline Architecture

1. Extract → 2. QA → 3. Join → 4. Enrich → 5. Deliver
Step What It Does On Failure
1. Extract Daily query from Snowflake. Pull Opportunities (Closed Lost, 2yr), junction records, usage snapshots. Store as raw immutable tables. If upstream SF → Snowflake extract is delayed, entire pipeline waits. Alert on SLA breach (>2hr past scheduled time). Serve stale data.
2. QA 6-check validation layer (see Section 6). Quality gate on critical checks. Critical fail: block load, serve last good, page on-call. Soft fail: log, flag rows, proceed.
3. Join LEFT JOIN Opportunity → Junction → Usage Snapshots. Point-in-time on Close_Date. Output: one row per deal-competitor. If snapshot table is empty (first run or backfill failure), all usage columns are NULL. Pipeline proceeds but dashboard shows “No usage data” for every account. Detectable via completeness check.
4. Enrich Derive: Usage_Score, Usage_Tier, Days_Inactive, Risk_Flag, Usage_Trend, Loss_Reason_Classified. If LLM service is down, Loss_Reason_Classified is NULL — fall back to primary picklist. All other metrics are pure SQL, no external dependency.
5. Deliver Write to competitive_deal_outcomes. Trigger Tableau extract refresh. Slack digest job. CS alert evaluation. If Tableau refresh fails, users see stale dashboard with “Last refreshed” timestamp. Slack digest skips if source table is stale (>48hr).
SLA target: <1 hour from Snowflake extract completion to dashboard refresh. Caveat: this assumes the upstream SF → Azure → Snowflake extract completes on schedule. We don’t control that pipeline. If upstream is delayed, our SLA breaks. Need to negotiate upstream SLA separately.
Security (embedded in pipeline): PII encrypted at rest (Competitive_Notes may contain customer names). Role-based access: CI team sees all, CS sees their accounts, Execs see aggregates. Audit log on every extraction. Minimum-permission service accounts for Airflow jobs.

Licensed_Seats

MEDIUM
Source
asset_line_item.quantity
Type
Integer
Valid Range
1 – 100,000 (cap at 100K)
SELECT account_id,
       MAX(quantity) AS licensed_seats
FROM asset_line_items
WHERE product_family = 'Sales Cloud'    -- filter to relevant product
  AND status = 'Active'                  -- exclude expired/cancelled assets
  AND quantity > 0                      -- exclude $0 line items
GROUP BY account_id
Edge case: multi-product accounts. An account can have Sales Cloud (50 seats) + Service Cloud (30 seats). Do we sum across products or take per-product? Per-product for now — each product line gets its own usage analysis. Cross-product composite is V2.
Edge case: expired assets. If an asset expired before close date, it shouldn’t count. Filter on status = 'Active' AND asset dates overlapping the close period.
NULL handling: If no asset line items exist for the account, Licensed_Seats = NULL. This is a signal — no tracked product, not “zero seats.” Do NOT default to 0.

Active_Users

MEDIUM
Source
usage.user_id, usage.event_ts
Type
Integer
Valid Range
0 – Licensed_Seats
SELECT account_id,
       COUNT(DISTINCT user_id) AS active_users
FROM usage_events
WHERE event_ts BETWEEN DATEADD('day', -30, :snapshot_date)
                   AND :snapshot_date
GROUP BY account_id
Open question: what counts as “active”? Login? Any feature use? API call? This definition changes the number dramatically. A login-only definition will show higher activity than a feature-use definition. Propose: start with login as baseline, add feature-use as a separate metric in V2.
Edge case: Active_Users > Licensed_Seats. Possible if licenses were downgraded but users still have cached sessions. Cap at Licensed_Seats in the ratio, log violations.
NULL handling: If no usage events exist for the account in the 30-day window, Active_Users = 0 (not NULL). Zero is meaningful — they have a product but nobody used it.

Usage_Score

MEDIUM

Weighted composite. The weights (50/30/20) are assumptions. Calibration against known churned vs retained accounts is a Week 2–3 task. If feature adoption data doesn’t exist, this becomes a 2-factor model with rebalanced weights (60/40).

Usage_Score = (w1 × License_Util) + (w2 × Active_User_Rate) + (w3 × Feature_Adoption)
Component Proposed Weight Calculation Example Confidence
License Utilization0.50Assigned / Licensed35/50 = 70%MED
Active User Rate0.30MAU / Assigned20/35 = 57%MED
Feature Adoption0.20Core_Feature_Users / MAU12/20 = 60%LOW
 
WITH base AS (
  SELECT
    a.account_id,
    a.licensed_seats,
    a.active_users,
    a.feature_users,
    -- License Utilization
    CASE WHEN a.licensed_seats > 0
         THEN a.active_users::FLOAT / a.licensed_seats
         ELSE NULL END AS license_util,
    -- Active User Rate
    CASE WHEN a.active_users > 0
         THEN LEAST(a.active_users::FLOAT / NULLIF(a.licensed_seats, 0), 1.0)
         ELSE 0.0 END AS active_user_rate,
    -- Feature Adoption (may be NULL if not tracked)
    CASE WHEN a.active_users > 0 AND a.feature_users IS NOT NULL
         THEN a.feature_users::FLOAT / a.active_users
         ELSE NULL END AS feature_adoption
  FROM account_usage_snapshot a
)
SELECT
  account_id,
  CASE
    -- 3-factor model (all components available)
    WHEN feature_adoption IS NOT NULL THEN
      (0.50 * COALESCE(license_util, 0))
    + (0.30 * active_user_rate)
    + (0.20 * feature_adoption)
    -- 2-factor fallback (feature adoption not tracked)
    WHEN license_util IS NOT NULL THEN
      (0.60 * license_util) + (0.40 * active_user_rate)
    -- No usage data at all
    ELSE NULL
  END AS usage_score
FROM base
Example (3-factor): (0.50 × 0.70) + (0.30 × 0.57) + (0.20 × 0.60) = 0.35 + 0.171 + 0.12 = 64.1%
Example (2-factor fallback): (0.60 × 0.70) + (0.40 × 0.57) = 0.42 + 0.228 = 64.8%

Usage_Tier

MEDIUM

Thresholds (30% / 70%) are assumptions based on industry benchmarks. Must calibrate against actual churn data. If the natural break points are at 25% and 60%, these thresholds are wrong and the dashboard misleads.

CASE
  WHEN usage_score IS NULL THEN 'NO_DATA'    -- don't default to LOW
  WHEN usage_score < 0.30  THEN 'LOW'
  WHEN usage_score < 0.70  THEN 'MED'
  ELSE 'HIGH'
END AS usage_tier
Critical: NULL usage ≠ LOW usage. An account with no tracking configured is different from an account with 10% utilization. The dashboard must show “No Data” as a separate state, not lumped into LOW.

Days_Inactive

MEDIUM 
CASE
  WHEN last_active IS NULL THEN NULL             -- no tracking = unknown, not 0
  WHEN last_active > close_date THEN 0          -- data quality issue: log but set to 0
  ELSE DATEDIFF('day', last_active, close_date)
END AS days_inactive
Edge case: Last_Active > Close_Date. Shouldn’t happen if point-in-time join is correct. If it does, it’s a data quality issue (snapshot date mismatch). Log to QA table, set to 0.
NULL handling: NULL = “we don’t know when they were last active.” Do NOT default to a large number (that would make them look inactive when we simply don’t have data).

Risk_Flag

MEDIUM 
CASE
  WHEN usage_tier = 'LOW'
   AND days_to_renewal IS NOT NULL
   AND days_to_renewal < 90
  THEN TRUE
  ELSE FALSE
END AS risk_flag
Edge case: renewal date unavailable. If renewal date isn’t on Opportunity, must derive from Contract end date. If neither exists, Risk_Flag = FALSE (we can’t assess timing risk). This is a data gap, not a clean bill of health — dashboard should show “Renewal date unknown.”
Threshold: 90 days is an assumption. Gives CS ~10 weeks to intervene. May need tuning — some enterprise renewals need 6-month lead time. Make configurable.

Usage_Trend (12-Month)

MEDIUM

Classifies the shape of usage decline over 12 months before close. A score of 18% at close means nothing without context — was it always 18% or did it used to be 70%?

WITH monthly_scores AS (
  SELECT
    account_id,
    DATE_TRUNC('month', snapshot_date) AS month,
    AVG(usage_score) AS avg_score,
    ROW_NUMBER() OVER (
      PARTITION BY account_id ORDER BY DATE_TRUNC('month', snapshot_date)
    ) AS month_idx
  FROM account_usage_snapshots
  WHERE snapshot_date BETWEEN DATEADD('month', -12, :close_date)
                          AND :close_date
  GROUP BY account_id, DATE_TRUNC('month', snapshot_date)
),
slopes AS (
  SELECT account_id,
    REGR_SLOPE(avg_score, month_idx) AS trend_slope,
    COUNT(*) AS months_available,
    STDDEV(avg_score) AS score_volatility
  FROM monthly_scores
  GROUP BY account_id
)
SELECT account_id,
  CASE
    WHEN months_available < 6          THEN 'INSUFFICIENT_DATA'
    WHEN trend_slope < -0.03
     AND score_volatility < 0.15      THEN 'PREDICTABLE_DECLINE'
    WHEN trend_slope < -0.08           THEN 'SUDDEN_DROP'
    WHEN ABS(trend_slope) < 0.01
     AND AVG(avg_score) < 0.30       THEN 'NEVER_ADOPTED'
    WHEN trend_slope > 0.02            THEN 'IMPROVING'
    ELSE 'STABLE'
  END AS usage_trend
FROM slopes
Slope thresholds (-0.03, -0.08, 0.01) are guesses. Need calibration against labeled examples of known decline patterns. These will shift when we see real data distributions. Make them configurable parameters, not hardcoded.
Edge case: <6 months of data. Trend classification is unreliable with sparse data. Return INSUFFICIENT_DATA — don’t force-classify.
Classification Pattern Implication
PREDICTABLE_DECLINE70% → 18% steady dropPreventable. CS should have intervened earlier.
SUDDEN_DROP65% → 10% in 2 monthsSpecific event. Champion left? Competitor demo? Investigate.
NEVER_ADOPTED~20% flat the entire timeOnboarding failure, not churn. Different root cause.
IMPROVINGUpward trendGood sign. Expansion target.
STABLENo significant changeMonitor. May be fine or may be stagnant.
INSUFFICIENT_DATA<6 months availableCannot classify. Do not guess.

09 Risks & Failure Modes

Risk Likelihood Impact Mitigation
Usage data doesn’t exist in expected form MEDIUM HIGH Day 1 discovery task. If asset hierarchy is different, redesign derivation. If no usage events exist, Usage_Score cannot be built — pipeline delivers deal + competitor data only (still valuable).
Historical snapshots don’t exist MEDIUM MEDIUM Build SCD Type 2 from scratch. Lose all historical trend analysis before build date. Backfill from any available monthly aggregates if they exist.
Competitive_Notes fill rate is low (<40%) MEDIUM MEDIUM Fall back to primary loss reason picklist. LLM classification deferred to V2. Still surface the data gap as a finding (“60% of lost deals have no competitive notes — process improvement needed”).
Leading flag not maintained post-close MEDIUM MEDIUM Cross-reference with Opportunity Stage history. If only one competitor on the deal and stage = Closed Lost, that competitor won by default. For multi-competitor deals without Leading flag, label as UNATTRIBUTED.
LLM classification produces garbage MEDIUM LOW LLM output is additive (separate column). Does not replace source data. If accuracy <70% on labeled sample, don’t ship to dashboard — keep in dev until improved.
908-record competitor catalog has duplicates HIGH MEDIUM Build entity resolution mapping table (e.g., “Oracle Cloud” → “Oracle”). Use in Enrich step. Existing zero-shot classification approach can be extended for this.
